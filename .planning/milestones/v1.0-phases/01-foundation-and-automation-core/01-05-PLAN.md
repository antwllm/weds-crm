1. Create `src/services/token-store.ts`:
   - OAuth refresh token persistence in the database to survive Cloud Run restarts.
   - Use a simple key-value approach: store tokens in a dedicated row in the database. Either:
     * Add a `oauthTokens` table to the schema (id, userId, accessToken, refreshToken, expiresAt, updatedAt) -- if adding to schema, update src/db/schema.ts to include this table
     * OR use a JSON column on an existing config/settings table
   - Preferred approach: add `oauthTokens` table to schema with columns: id (serial PK), email (text, unique -- the authenticated user's email), accessToken (text), refreshToken (text), expiresAt (timestamp, nullable), updatedAt (timestamp, default now())
   - `saveTokens(email: string, accessToken: string, refreshToken: string, expiresAt?: Date): Promise<void>` -- upsert (INSERT ON CONFLICT UPDATE) tokens for the given email
   - `loadTokens(email: string): Promise<{ accessToken: string, refreshToken: string, expiresAt?: Date } | null>` -- retrieve stored tokens for the given email, return null if not found
   - Export saveTokens and loadTokens
   - Update `src/auth/passport.ts` verify callback: after successful authentication, call saveTokens to persist the refresh token in DB
   - Update `src/db/schema.ts`: add oauthTokens table definition

2. Create `src/services/pubsub.ts`:
   - `handlePubSubMessage(messageData: string): { historyId: string, emailAddress: string }` -- decodes base64 Pub/Sub message data and extracts Gmail notification fields
   - Validates that emailAddress matches the configured allowed user email
   - Returns parsed data for the webhook handler to use

3. Create `src/pipeline/scheduler.ts`:
   - `startScheduler(gmail: gmail_v1.Gmail): void`
   - Uses node-cron to schedule two jobs:
     * Fallback sweep: config.fallbackSweepInterval (default '*/30 * * * *') -> calls processPendingEmails(gmail)
     * Gmail watch renewal: '0 3 * * *' (daily at 3 AM) -> calls renewGmailWatch
   - `renewGmailWatch(gmail: gmail_v1.Gmail): Promise<void>` -- calls gmail.users.watch with the configured Pub/Sub topic and label filter. Logs expiration timestamp.
   - Renew watch immediately on scheduler start (not just on cron)
   - Export startScheduler

4. Create `src/routes/webhook.ts`: Express Router with:
   - POST `/webhook/gmail` -- receives Pub/Sub push notification
   - Extracts message.data from req.body, decodes with handlePubSubMessage
   - Calls processPendingEmails (do NOT process based on historyId -- just trigger a full sweep, which is simpler and the concurrency guard protects against redundant runs)
   - Returns 200 immediately (acknowledge Pub/Sub before processing completes to avoid redelivery -- use setImmediate or process.nextTick for async processing)
   - No auth required on this endpoint (Pub/Sub push uses OIDC verification at the Cloud Run level, not in app code)

5. Update `src/app.ts`:
   - Mount webhook route: app.use('/webhook', webhookRouter)
   - After successful OAuth callback, store tokens via saveTokens and initialize Gmail client
   - Add a mechanism to get the authenticated Gmail client for the pipeline: store the Gmail client instance in a module-level singleton (`src/services/gmail-client-holder.ts` or directly in app.ts via a `getGmailClientInstance()` export)

6. Update `src/index.ts`:
   - After server starts, attempt to restore Gmail client from stored refresh token by calling loadTokens(config.allowedUserEmail)
   - If tokens found: create Gmail client using getGmailClient(accessToken, refreshToken), start scheduler, renew watch
   - If no tokens found: log "En attente d'authentification OAuth -- connectez-vous via /auth/google" (French) -- scheduler starts after first login
