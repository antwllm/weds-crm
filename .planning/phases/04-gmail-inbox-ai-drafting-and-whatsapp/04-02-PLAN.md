<task type="auto" tdd="true">
  <name>Task 1: Gmail inbox API routes + email-to-lead linking</name>
  <files>src/routes/api/inbox.ts, tests/api/inbox.test.ts</files>
  <behavior>
    - GET /api/inbox/threads?pageToken=X&q=Y returns { threads: [...], nextPageToken } from Gmail
    - GET /api/inbox/threads/:threadId returns full thread with messages, each message has from, to, subject, date, body, messageId
    - GET /api/inbox/threads/:threadId also returns linked lead info if sender email matches a lead
    - POST /api/inbox/threads/:threadId/reply with { to, subject, body, inReplyTo, references } sends reply and creates linkedEmail + email_sent activity
    - GET /api/leads/:leadId/emails returns all linkedEmails for a lead, ordered by receivedAt desc
    - POST /api/inbox/link-email with { gmailMessageId, gmailThreadId, leadId, subject, snippet, direction, receivedAt } creates a linkedEmail record
  </behavior>
  <action>
1. Create src/routes/api/inbox.ts:
   - Import ensureAuthenticated, getGmailClientInstance, listThreads, getThread, sendReply from services
   - All routes use ensureAuthenticated middleware

   GET /inbox/threads:
   - Get gmail client via getGmailClientInstance(), return 503 if null
   - Call listThreads(gmail, { maxResults: req.query.maxResults || 20, pageToken: req.query.pageToken, q: req.query.q || 'in:inbox' })
   - For each thread in result, also query linkedEmails + leads to check if any message sender matches a lead's email (lowercase comparison)
   - Return { threads: [{ id, snippet, historyId, matchedLead?: { id, name, status } }], nextPageToken }

   GET /inbox/threads/:threadId:
   - Call getThread(gmail, threadId)
   - For each message sender, check if email matches a lead (extract email from "Name <email>" format, lowercase)
   - If lead found, include lead info and auto-link: insert into linkedEmails if not already linked (upsert by gmailMessageId)
   - Return thread with messages + linked lead info. This fulfills MAIL-04 (auto-linking).

   POST /inbox/threads/:threadId/reply:
   - Zod validate body: { to: string, subject: string, body: string, inReplyTo: string, references: string }
   - Call sendReply(gmail, { threadId, to, subject, body, inReplyTo, references })
   - Insert into linkedEmails: { leadId (if matched), gmailMessageId: 'sent-' + Date.now(), gmailThreadId: threadId, subject, snippet: body.slice(0, 200), direction: 'outbound', receivedAt: new Date() }
   - If leadId matched, create email_sent activity
   - Return 200 { status: 'sent' }

   GET /leads/:leadId/emails:
   - Query linkedEmails where leadId matches, order by receivedAt desc
   - Return { emails: [...] }

2. Mount in app.ts: import inboxRouter, app.use('/api', inboxRouter)

3. Write tests/api/inbox.test.ts:
   - Mock getGmailClientInstance, listThreads, getThread, sendReply
   - Test GET /api/inbox/threads returns thread list
   - Test GET /api/inbox/threads/:id returns thread detail with lead matching
   - Test POST /api/inbox/threads/:id/reply calls sendReply with correct params
   - Test GET /api/leads/:id/emails returns linked emails
  </action>
  <verify>
    <automated>npx vitest run tests/api/inbox.test.ts -x --reporter=verbose</automated>
  </verify>
  <done>All 4 inbox endpoints work: list threads, get thread detail with auto-linking, send reply, get lead emails. Tests pass.</done>

  <files>src/routes/api/templates.ts, src/routes/api/ai.ts, src/app.ts, tests/api/templates.test.ts, tests/api/ai.test.ts</files>
  <behavior>
    - GET /api/templates returns all email templates
    - POST /api/templates creates a template with name, subject, body, variables
    - PUT /api/templates/:id updates a template
    - DELETE /api/templates/:id deletes a template
    - GET /api/ai/prompt returns current AI prompt config (or default if none exists)
    - PUT /api/ai/prompt updates AI prompt template and model
    - POST /api/ai/generate-draft with { leadId } returns { draft: string } -- proxies through OpenRouter server-side (never exposes API key to frontend)
    - POST /api/templates/:id/preview with { leadId } returns { subject, body } with variables substituted from lead data
  </behavior>
  <action>
1. Create src/routes/api/templates.ts:
   - GET /templates: select all from emailTemplates, order by createdAt desc
   - POST /templates: Zod validate { name: string, subject: string, body: string, variables?: string[] }. Insert into emailTemplates. Return 201 with created template.
   - PUT /templates/:id: Zod validate partial { name?, subject?, body?, variables? }. Update emailTemplates where id. Return updated template.
   - DELETE /templates/:id: delete from emailTemplates where id. Return 204.
   - POST /templates/:id/preview: Zod validate { leadId: number }. Fetch template + lead. Call substituteVariables on subject and body. Return { subject, body } with real lead data substituted.

2. Create src/routes/api/ai.ts:
   - GET /ai/prompt: select first row from aiPromptConfig. If none exists, return default { promptTemplate: 'Tu es un photographe de mariage professionnel...', model: 'anthropic/claude-sonnet-4' }.
   - PUT /ai/prompt: Zod validate { promptTemplate: string, model?: string }. Upsert into aiPromptConfig (if row exists update, else insert). Return updated config.
   - POST /ai/generate-draft: Zod validate { leadId: number }. Call assembleLeadContext(leadId). Fetch current AI prompt config. Call generateDraft(context, promptTemplate, config.OPENROUTER_API_KEY). Return { draft: string }. If OPENROUTER_API_KEY not configured, return 503 with message.
   - CRITICAL: The draft is returned as text to the frontend. It is NOT sent as an email. The frontend must display it in the compose window for review. There is no auto-send path.

3. Mount both routers in app.ts: app.use('/api', templatesRouter), app.use('/api', aiRouter)

4. Write tests:
   - tests/api/templates.test.ts: Test CRUD operations, test preview substitution
   - tests/api/ai.test.ts: Test prompt CRUD, test generate-draft calls OpenRouter (mock), test 503 when no API key
